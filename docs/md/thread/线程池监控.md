# 线程池监控

ThreadPoolExecutor 构造函数几个核心参数：

```java
corePoolSize：线程池中核心线程数的最大值

maximumPoolSize：线程池中能拥有最多线程数

keepAliveTime：空闲线程的存活时间

workQueue：用于缓存任务的阻塞队列

handler：拒绝策略
```

<img src="../../assets/img/%E7%BA%BF%E7%A8%8B%E6%B1%A0.png" alt="线程池" style="zoom:50%;" />



从上图进行简单分析，有三个核心的内容，线程池 ThreadPoolExecutor、队列 workQueue、拒绝策略 RejectedExecutionHandler

1. 线程池 ThreadPoolExecutor
   1. 核心线程数：corePoolSize
   2. 最大线程数：maximumPoolSize
   3. 线程池当前线程数：poolSize
   4. 活跃线程数：activeCount
   5. 同时进入池中的最大线程数：largestPoolSize
   6. 线程池中执行任务总数量：completedTaskCount
2. 队列 workQueue
   1. 拒绝策略：rejectedExecutionHandlerName
   2. 队列元素个数：queueSize
   3. 队列剩余容量：remainingCapacity
   4. 队列容量：queueCapacity
3. 拒绝策略 RejectedExecutionHandler
   1. 拒绝次数

```java
    /**
     * 线程池活跃度 = 线程池中线程数 / 线程池最大线程数
     */
    ACTIVITY("activity"),

    /**
     * 队列容量 = 队列中任务数 / 队列最大任务数
     */
    CAPACITY("capacity"),

    /**
     * 拒绝次数
     */
    REJECT("reject"),
```



接着往下分析：

线程池中任务存在2个状态，一个是执行，另一个是排队，拒绝属于拒绝策略，那么就存在以下2个属性：

1. 执行时间
2. 队列等待时间

虽然这两个属性严格意义上对于线程池来讲没有任何特殊操作，但是在我们实际的业务上面，单个线程的执行情况可能会更加直观的暴露问题，例如：定时微信查单，可以通过执行时间判断网络通信状况。

```java
    /**
     * 任务执行超时次数
     */
    RUN_TIMEOUT("run_timeout"),

    /**
     * 队列排队时间过长次数
     */
    QUEUE_TIMEOUT("queue_timeout");
```
